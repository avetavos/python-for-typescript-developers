## Coming from JavaScript's single-threaded world

JavaScript runs in a single thread. There is one call stack, one event loop, and no shared memory between concurrent units of work. When you write `async/await` in Node.js you are scheduling microtasks on that single thread — you never run two things truly at the same time inside one process.

Python gives you **three different concurrency tools** and you have to choose the right one for the job:

| Tool | What it is | Best for |
|---|---|---|
| `asyncio` | Single-threaded cooperative concurrency | I/O-bound async code (HTTP, DB, files) |
| `threading` | OS threads sharing one interpreter | I/O-bound code, legacy blocking libraries |
| `multiprocessing` | Separate OS processes | CPU-bound work that needs true parallelism |

## Concurrency vs parallelism

These two words are often conflated. They mean different things:

- **Concurrency** — multiple tasks are *in progress* at the same time, but not necessarily running at the exact same instant. The event loop switches between tasks when one is waiting. This is how Node.js works, and how Python's `asyncio` works.
- **Parallelism** — multiple tasks *execute* at the exact same instant on different CPU cores. This requires multiple OS processes in Python (or multiple threads in most other languages).

export const concurrencyTs = `// JavaScript/Node.js
// Concurrency via the event loop — single thread, never truly parallel
async function fetchUser(id: number): Promise<User> {
  // While this awaits, other promises can run
  return await db.query("SELECT * FROM users WHERE id = $1", [id]);
}

// Promise.all is concurrent, not parallel — still one thread
const [user, posts] = await Promise.all([
  fetchUser(1),
  fetchPosts(1),
]);`;

export const concurrencyPy = `# Python
# asyncio — same single-threaded cooperative model as JS
import asyncio

async def fetch_user(user_id: int) -> dict:
    # While this awaits, other coroutines can run
    await asyncio.sleep(0)  # yield to event loop
    return {"id": user_id}

# asyncio.gather — same idea as Promise.all
async def main() -> None:
    user, posts = await asyncio.gather(
        fetch_user(1),
        fetch_posts(1),
    )`;

<PyOnly title="The GIL — Python's most important concurrency constraint">

The **Global Interpreter Lock (GIL)** is a mutex inside CPython (the standard Python interpreter) that ensures only one thread executes Python bytecode at a time.

This has one major consequence for threading:

- **I/O-bound threads work fine.** While one thread is waiting for a network response, the GIL is released and another thread can run.
- **CPU-bound threads do NOT speed up.** Two threads crunching numbers will not use two cores — the GIL forces them to take turns.

JavaScript developers already live with this constraint (one thread), but it can be surprising when you expect Python threads to parallelize CPU work like Java or Go threads would.

The escape hatch is `multiprocessing`: each process has its own GIL (and its own memory), so CPU-bound work can run on all available cores.

```python
# GIL is released during I/O → threads CAN overlap here
import threading, urllib.request

def download(url: str) -> None:
    urllib.request.urlopen(url)  # GIL released while waiting

# GIL is NOT released during pure Python computation
def compute(n: int) -> int:
    return sum(range(n))  # threads here won't run in parallel
```

Python 3.13 introduced an **experimental no-GIL build** (PEP 703). It is not the default yet, but the direction is clear.
