## การย่อหน้าคือ syntax ไม่ใช่แค่ลวดลาย

ใน TypeScript การย่อหน้าเป็นเรื่องของรูปลักษณ์ linter บังคับใช้เพื่อให้อ่านง่าย แต่ compiler ไม่สนใจเลย code ทำงานเหมือนกันไม่ว่าจะไม่มีการย่อหน้าหรือเยื้องไม่สม่ำเสมอ ใน Python การย่อหน้าเป็นส่วนหนึ่งของ grammar interpreter ใช้มันเพื่อตัดสินว่า block เริ่มและสิ้นสุดที่ไหน ไม่มี curly braces และไม่มีคำสั่ง `end`

นี่ไม่ใช่ความแปลกประหลาด แต่เป็นการตัดสินใจออกแบบโดยเจตนา เพื่อบังคับให้ code อ่านง่ายโดยอัตโนมัติ

## บล็อก if, for และ function

export const tsWhitespace = `// TypeScript: braces delimit every block
function classify(n: number): string {
  if (n > 0) {
    return "positive";
  } else if (n < 0) {
    return "negative";
  } else {
    return "zero";
  }
}

for (let i = 0; i < 3; i++) {
  if (i % 2 === 0) {
    console.log(\`\${i} is even\`);
  }
}`;

export const pyWhitespace = `# Python: indentation delimits every block
def classify(n: int) -> str:
    if n > 0:
        return "positive"
    elif n < 0:
        return "negative"
    else:
        return "zero"

for i in range(3):
    if i % 2 == 0:
        print(f"\${i} is even")`;

<TsGo ts={tsWhitespace} go={pyWhitespace} />

ข้อแตกต่างด้าน syntax ที่ควรสังเกต:
- คำว่า `function` กลายเป็น `def` ตามด้วยชื่อฟังก์ชันและเครื่องหมายโคลอน
- `else if` ใน TypeScript กลายเป็น `elif` ใน Python (ไม่มี `else if` แยกต่างหาก)
- ทุกบรรทัดที่เปิด block ต้องลงท้ายด้วยเครื่องหมายโคลอน (`:`)
- มาตรฐานการย่อหน้าคือ **4 ช่องว่าง** การใช้ tab ได้แต่การผสม tab กับ space จะทำให้เกิด `TabError`

## IndentationError: error ที่ผู้เริ่มต้นเจอบ่อยที่สุด

ถ้าย่อหน้าผิด Python จะ raise `IndentationError` ก่อนที่ code จะรันด้วยซ้ำ นี่คือเทียบเท่ากับ error "unexpected token" ของ TypeScript ตอน compile

export const tsIndentTs = `// TypeScript: this still runs (just ugly style)
function add(a: number, b: number) {
return a + b; // no indent — TS doesn't care
}`;

export const pyIndentPy = `# Python: this raises IndentationError
def add(a: int, b: int) -> int:
return a + b  # missing indent — SyntaxError at parse time

# Correct version:
def add(a: int, b: int) -> int:
    return a + b  # 4 spaces`;

## บล็อกซ้อนกัน

การซ้อนบล็อกทำได้โดยเพิ่มระดับการย่อหน้าสำหรับแต่ละบล็อก Python parser ติดตามระดับการย่อหน้า จึงไม่มีขีดจำกัดในการซ้อน แต่ Zen of Python แนะนำให้รักษาระดับความลึกให้น้อยที่สุด

export const tsNestedWhitespace = `// TypeScript nested blocks
function processMatrix(matrix: number[][]): void {
  for (const row of matrix) {
    for (const cell of row) {
      if (cell > 0) {
        console.log(\`positive: \${cell}\`);
      }
    }
  }
}`;

export const pyNestedWhitespace = `# Python nested blocks
def process_matrix(matrix: list[list[int]]) -> None:
    for row in matrix:
        for cell in row:
            if cell > 0:
                print(f"positive: \${cell}")`;

## ลองด้วยตัวเอง

export const playgroundWhitespace = `# Python uses indentation — no curly braces needed
def greet(name):
    if name:
        print(f"Hello, {name}!")
    else:
        print("Hello, stranger!")

greet("Alice")
greet("")

for i in range(3):
    if i % 2 == 0:
        print(f"{i} is even")
    else:
        print(f"{i} is odd")
`;

<PyOnly title="Zen of Python: ความอ่านง่ายนับ">

การเลือกให้ whitespace มีความหมายไม่ใช่เรื่องบังเอิญ Guido van Rossum ทำให้มันเป็นส่วนหนึ่งของ identity ของ Python และสรุปไว้ในบรรทัดที่สองของ Zen of Python:

> Readability counts.

ในภาษาสไตล์ C ส่วนใหญ่ ทีมต้องตกลงกันเรื่อง brace style (Allman, K&R, 1TBS) และบังคับใช้ผ่าน linter ใน Python ไม่มีอะไรให้ถกเถียง ทุกไฟล์ Python ใช้ตัวกำหนด block เดียวกัน คือ โคลอนและสี่ช่องว่าง code ที่เขียนโดยนักพัฒนาต่างกัน ต่างทีม ต่างบริษัท จะมีโครงสร้างที่เหมือนกันทุกประการ

สำหรับนักพัฒนา TypeScript การเปลี่ยน mindset ที่ยากที่สุดคือการยอมรับว่าการไม่มี braces ไม่ใช่ข้อจำกัด แต่เป็นคุณสมบัติ คุณจะพบว่า code Python อ่านได้เร็วกว่า เพราะตาไม่ต้องกรองเครื่องหมาย `{` และ `}` ที่วางอยู่บนบรรทัดของตัวเอง

รัน `import this` ใน Python interpreter ใดก็ได้เพื่ออ่าน Zen of Python ฉบับเต็ม บรรทัด "There should be one — and preferably only one — obvious way to do it" อธิบายว่าทำไม Python ถึงทำให้การย่อหน้าเป็นตัวเลือกเดียว ไม่ใช่หนึ่งในหลายทางเลือก
